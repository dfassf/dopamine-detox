# Phase 1: 회원가입 + 로그인

## 개요

사용자가 앱의 핵심 기능(금욕 타임라인)에 접근하기 위한 인증 터널.
가입 허들을 최소화하되, 개인화에 필요한 식별 정보를 확보한다.

---

## 인증 방식

### MVP

| 항목 | 내용 |
|---|---|
| 방식 | 이메일 + 비밀번호 |
| 토큰 | JWT (Access Token + Refresh Token) |
| 저장 | Access Token → 메모리, Refresh Token → httpOnly Cookie |

### 추후 (출시 검토 시)

- 소셜 로그인(구글/카카오)으로 교체
- 이메일+비밀번호 방식은 제거

> MVP에서 소셜 로그인을 고려한 설계를 할 필요는 없다.
> 교체 시점에 인증 레이어를 통째로 바꾸면 된다.

---

## 회원가입

### 수집 정보

| 필드 | 타입 | 필수 | 용도 |
|---|---|---|---|
| 이메일 | string (email) | O | 로그인 식별자 |
| 비밀번호 | string | O | 인증 |
| 닉네임 | string | O | 앱 내 표시 이름 |

> **나이, 성별, 체중 등 개인화 데이터는 가입 시 받지 않는다.**
> → Phase 2의 "금욕 등록" 단계에서 수집.
> → 가입 허들을 낮추기 위함.

### 가입 플로우

```
[랜딩 페이지]
    │
    ├─ "시작하기" 클릭
    │
    ▼
[회원가입 폼]
    │  이메일
    │  비밀번호
    │  비밀번호 확인
    │  닉네임
    │
    ├─ 가입 완료
    │
    ▼
[로그인 상태로 전환]
    │
    ▼
[메인 화면 (금욕 등록 유도)]
```

### 유효성 검증

| 필드 | 프론트엔드 | 백엔드 |
|---|---|---|
| 이메일 | 형식 검증 | 형식 검증 + 중복 확인 |
| 비밀번호 | 8자 이상 | 8자 이상, bcrypt 해싱 저장 |
| 비밀번호 확인 | 일치 여부 | — (프론트에서만 처리) |
| 닉네임 | 2~20자 | 2~20자, 공백 트림 |

### 가입 API

```
POST /api/auth/signup

Request:
{
  "email": "user@example.com",
  "password": "securepassword",
  "nickname": "금주왕"
}

Response (201):
{
  "access_token": "eyJ...",
  "user": {
    "id": 1,
    "email": "user@example.com",
    "nickname": "금주왕"
  }
}

Error (409):
{
  "detail": "이미 가입된 이메일입니다"
}
```

## 로그인

### 로그인 플로우

```
[로그인 폼]
    │  이메일
    │  비밀번호
    │
    ├─ 인증 성공
    │
    ▼
[메인 화면]

    ├─ 인증 실패
    │
    ▼
[에러 메시지: "이메일 또는 비밀번호가 올바르지 않습니다"]
```

> 보안상 이메일/비밀번호 중 어느 쪽이 틀렸는지 구분하지 않는다.

### 로그인 API

```
POST /api/auth/login

Request:
{
  "email": "user@example.com",
  "password": "securepassword"
}

Response (200):
{
  "access_token": "eyJ...",
  "user": {
    "id": 1,
    "email": "user@example.com",
    "nickname": "금주왕"
  }
}

Error (401):
{
  "detail": "이메일 또는 비밀번호가 올바르지 않습니다"
}
```

## 토큰 관리

### JWT 구조

```
Access Token:
{
  "sub": 1,          // user id
  "exp": 1700000000  // 만료 시간
}

유효 기간: 30분
```

```
Refresh Token:
{
  "sub": 1,
  "exp": 1700000000
}

유효 기간: 7일
저장: httpOnly Cookie
```

### 토큰 갱신 API

```
POST /api/auth/refresh

Request: (Cookie에서 Refresh Token 자동 전송)

Response (200):
{
  "access_token": "eyJ..."
}

Error (401):
{
  "detail": "로그인이 만료되었습니다"
}
```

### 로그아웃 API

```
POST /api/auth/logout

Response (200):
{
  "message": "로그아웃 되었습니다"
}

→ Refresh Token Cookie 삭제
```

## 인증 미들웨어

로그인이 필요한 모든 API에 적용.

```
Authorization: Bearer {access_token}
```

| 상황 | 동작 |
|---|---|
| 유효한 토큰 | 요청 처리 |
| 만료된 Access Token | 401 → 프론트에서 refresh 호출 후 재시도 |
| 만료된 Refresh Token | 401 → 로그인 페이지로 이동 |

## 비로그인 상태 (터널 바깥)

비로그인 사용자가 볼 수 있는 것:

| 페이지 | 내용 |
|---|---|
| 랜딩 페이지 | 앱 소개 + "시작하기" 버튼 |
| 로그인 페이지 | 이메일/비밀번호 입력 |
| 회원가입 페이지 | 가입 폼 |

> 그 외 모든 페이지는 로그인 필요.
> 비로그인 상태에서 접근 시 로그인 페이지로 리다이렉트.
